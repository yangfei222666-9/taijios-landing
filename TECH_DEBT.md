# 官网技术债清单 · 2026-04-24

状态: 首页、经验池、报价页已完成新定位改版。当前技术债重点不是继续美化, 而是清掉会造成旧叙事、旧承诺、构建不可复现的问题。

## P0 · 会影响对外可信度

1. `about.md` 仍是旧叙事入口
   - 风险: 可能继续暴露旧的诸葛孔明、足球预测、自进化夸大口径。
   - 最小动作: 改成中文「关于太极OS」, 只讲智能代理工作流可靠性、事件流、经验池。

2. `install.md` 仍是旧安装页
   - 风险: 新用户会从官网进入旧路线, 与现在的服务报价和可靠性定位断层。
   - 最小动作: 改成「开始接入」页, 引导用户提交当前自动化链路、失败样本和目标输出。

3. `_layouts/post.html` 和 `blog.html` 仍保留旧博客框架
   - 风险: 新首页很强, 但点击笔记后会回到旧视觉和旧信息架构。
   - 最小动作: 把博客标成「工程日志」, 旧文章加 archive 语义, 不再像主卖点。

4. `og.png` 可能仍是旧宣传图
   - 风险: X/Telegram 分享时继续展示旧口径, 抵消官网改版。
   - 最小动作: 重新生成 OG 图, 文案改为「让智能代理失败会主动报警」。

## P1 · 会造成后续漂移

1. 首页、经验池、报价页的 CSS 和导航重复
   - 风险: 每次改导航、颜色、CTA 都要改多处, 很容易再次 drift。
   - 最小动作: 抽出共享 header/footer 和基础 CSS, 或至少建立一份页面模板。

2. 本地构建不可复现
   - 风险: 现在主要靠静态检查和 GitHub Pages 验证, 本地 `bundle exec jekyll build` 依赖不完整时不能快速发现问题。
   - 最小动作: 补 `Gemfile.lock` 或 `bin/check_site`, 明确本地验证命令。

3. 链接检查缺失
   - 风险: `/pricing/`、`/crystals/`、`/install/`、`/blog.html` 任一链接坏掉, 只能上线后人工发现。
   - 最小动作: 加一个轻量脚本检查站内链接和关键文案。

## P2 · 可延后但要记录

1. 旧英文文章需要重新分类
   - 风险: 旧内容不是不能留, 但不能让它们代表当前太极OS主张。
   - 最小动作: 建 archive 标识, 在旧文章顶部注明「历史记录, 非当前服务承诺」。

2. 联系表单缺失
   - 风险: 当前靠 Telegram/Email/微信, 转化线索不可结构化。
   - 最小动作: 加一个最小表单或 mailto 模板, 固定收集「已有链路 / 失败样本 / 每日目标输出」。

3. 事件流卖点还没有交互演示
   - 风险: 现在讲得清楚, 但没有 demo 会降低可信度。
   - 最小动作: 做一个静态「失败事件流示例」区块, 展示 ok / degraded / failed 的差异。

## 执行顺序

1. 先改 `about.md` 和 `install.md`, 避免用户从旧入口进入。
2. 再重做 `og.png`, 保证社交分享不打架。
3. 再抽共享模板和检查脚本, 降低后续维护成本。
4. 最后处理旧博客归档和事件流演示。
