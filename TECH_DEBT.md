## P1 · 会造成后续漂移

1. 多页面 CSS 和导航重复
   - 风险: 每次改导航、颜色、CTA 都要改多处, 很容易再次 drift。
   - 最小动作: 抽出共享 header/footer 和基础 CSS, 或至少建立一份页面模板。

2. 本地构建不可复现
   - 风险: 现在主要靠静态检查和 GitHub Pages 验证, 本地 `bundle exec jekyll build` 依赖不完整时不能快速发现问题。
   - 最小动作: 补 `Gemfile.lock` 或 `bin/check_site`, 明确本地验证命令。

3. 链接检查缺失
   - 风险: `/pricing/`、`/crystals/`、`/install/`、`/blog.html` 任一链接坏掉, 只能上线后人工发现。
   - 最小动作: 加一个轻量脚本检查站内链接和关键文案。

## P2 · 可延后但要记录

1. 联系表单缺失
   - 风险: 当前靠 Telegram/Email/微信, 转化线索不可结构化。
   - 最小动作: 加一个最小表单或 mailto 模板, 固定收集「已有链路 / 失败样本 / 每日目标输出」。

2. 事件流卖点还没有交互演示
   - 风险: 现在讲得清楚, 但没有 demo 会降低可信度。
   - 最小动作: 做一个静态「失败事件流示例」区块, 展示 ok / degraded / failed 的差异。

## 执行顺序

1. 先补本地构建/检查脚本, 让每次上线前能一键检查关键路由和关键文案。
2. 再抽共享模板和基础 CSS, 降低导航与视觉漂移。
3. 再补联系表单或 mailto 模板, 结构化线索。
4. 最后做事件流交互演示。
